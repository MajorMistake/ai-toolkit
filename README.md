Personal AI skills, prompts, and accumulated knowledge. General-purpose methodology only — no employer-specific content.

## Design decisions

### Why `skills/` follows the SKILL.md convention exactly
Each subfolder is a standalone skill that can be copied or symlinked directly into `~/.claude/skills/` for Claude Code, or uploaded to claude.ai. No reformatting needed. The folder structure matches what Claude expects — just move it into place.

### Why `prompts/` is separate from `skills/`
Skills are model-invoked — Claude decides when to use them based on context. Prompts are human-invoked — you paste them or reference them when you need them. Different trigger mechanism, different design constraints. A skill needs a good description so Claude triggers it correctly. A prompt needs to be easy for *you* to find and adapt.

### Why `learnings/` instead of a single LEARNINGS.md at root
LEARNINGS.md is the Copilot convention and should stay as-is for compatibility. But you'll accumulate other knowledge that doesn't fit that format — patterns you've noticed, mistakes worth remembering, mental models that improved your work. Giving these their own files keeps LEARNINGS.md clean for Copilot while giving you space to capture broader knowledge.

### Why `system-prompts/` exists
You use multiple AI tools (claude.ai, Claude Code, Claude Desktop, Copilot) and each has its own custom instructions mechanism. Keeping these version-controlled means: you can track how your preferences evolve, you can diff what you tell different tools, and you don't lose them when you switch machines or reset a tool. This is also where your claude.ai user preferences live as source-of-truth — if you update them, you update the file here first, then paste into settings.

### Why `setup/` exists
Future-you (or anyone you share a subset with) needs to know how to actually wire these files into the right tools. This is the "installation manual." Symlink commands, where Claude Code looks for skills, how to load a CLAUDE.md, etc.

## Usage patterns

### Adding a new skill
1. Create a new folder under `skills/` with a `SKILL.md`
2. Follow the frontmatter convention (name, description in YAML)
3. Commit and push
4. On any machine: symlink or copy into `~/.claude/skills/`

### Adding a new prompt
1. Create a markdown file under `prompts/`
2. Add an entry to `prompts/README.md` describing when to use it
3. When you need it: open the file, copy, paste into your AI tool

### Updating learnings
LEARNINGS.md can be regenerated by Copilot, but anything you manually curate in patterns.md or mistakes.md should be treated as your own writing — edit directly.

### Syncing system prompts
Edit the file in `system-prompts/`, commit, then paste the updated content into the relevant tool's settings. The repo is the source of truth; the tool settings are the deployment target.

## What doesn't go here
- Employer-specific configurations, team names, internal URLs
- API keys, tokens, credentials of any kind
- Content generated from proprietary data
- Skills that reference internal tools or systems by name

If a skill needs employer-specific context to work, keep a generic version here and a customized fork in your work environment.
